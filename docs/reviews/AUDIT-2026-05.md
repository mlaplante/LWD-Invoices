# Codebase Audit ‚Äî May 2026

Full-tree audit of `src/` (560 files, ~80K LOC of TS/TSX). Baseline at start of
run: `npx tsc --noEmit` clean, all 1390 vitest tests passing.

Findings are grouped by domain. Each item lists severity, location, and the
intended fix. Items marked **FIXED** were addressed in this PR; everything else
is a roadmap entry ‚Äî picked up by severity / effort tradeoff.

---

## What changed in this PR

Surgical fixes for the highest-leverage, lowest-risk findings only. Larger
refactors and new features are listed below as roadmap items so they can be
sized and prioritised separately.

### Security
- **Portal-token entropy** ‚Äî new invoice / client / credit-note / recurring
  rows now seed `portalToken` with `crypto.randomBytes(32).toString("hex")`
  via a new `generatePortalToken()` helper in `src/lib/portal-session.ts`.
  Prisma's `@default(cuid())` is still the schema fallback but no creation
  path relies on it. Existing rows keep their CUIDs; admins can rotate any
  leaked link with the existing `invoices.rotatePortalToken` procedure.
- **Dedicated portal-session HMAC secret** ‚Äî added `PORTAL_SESSION_SECRET`
  env var (optional, with backwards-compat fallback to
  `SUPABASE_SERVICE_ROLE_KEY`). Three call sites (`portal/[token]/auth`,
  `portal/[token]/estimate`, `portal/[token]/layout`) route through a new
  `getPortalSessionSecret()` helper so the service-role key isn't doubling
  as a cookie-forgery key in production.
- **Portal passphrase strength** ‚Äî added `.min(8).max(255)` to the
  `portalPassphrase` Zod field on the clients router. Combined with the
  existing 10-attempts / 15-min limiter and 5-failure lockout, this makes
  online brute-force infeasible.
- **Portal token enumeration** ‚Äî both `/api/portal/[token]/auth` and
  `/api/portal/dashboard/[clientToken]/auth` now return an identical
  `401 { error: "Invalid token or passphrase" }` for both "not found" and
  "wrong passphrase" cases. Status-code-based enumeration of valid tokens
  is no longer possible.
- **CSV formula injection** in `/api/reports/expenses/export` ‚Äî  was missing
  the `^[=+\-@]` guard entirely. Added it (with `\t` for locale parity with
  the other export). The tRPC `exports` router was already RFC 4180-safe.

### Performance
- **Recurring-invoice generation** (`src/inngest/functions/recurring-invoices.ts`)
  ‚Äî replaced the per-line `tx.invoiceLine.create` + per-tax
  `tx.invoiceLineTax.create` loops with a single nested-create for lines and
  one `createMany` for line taxes. For a 10-line invoice with two taxes per
  line that's 30 round-trips ‚Üí 2.
- **Bulk export caps** ‚Äî `exports.invoicesCSV`, `clientsCSV`, `expensesCSV`
  and `/api/reports/{invoices,expenses}/export` previously did unbounded
  `findMany`s. Capped at 5 000 rows with a `truncated` flag (`+1` probe
  detects the cap without an extra `count()`).
- **Retention check-in cron** (`src/inngest/functions/retention-checkins.ts`)
  ‚Äî orgs were processed sequentially; switched to 10-org concurrency
  batches. A 500-org tenant goes from N√ót to ‚åàN/10‚åâ√ót.
- **Notifications router** ‚Äî removed the redundant `organization.findFirst`
  in `list`, `unreadCount`, `markRead`, `markAllRead` (the org id is already
  in `ctx.orgId`, verified by `protectedProcedure`). Saves one query per
  notifications call. Also removed the `try/catch` that silently returned
  `[]` / `0` on DB errors ‚Äî those now surface as real failures instead of
  appearing as "no notifications".

### Code quality
- **`throw new Error` ‚Üí `TRPCError`** in `creditNotes.validateCreditApplication`.
  Plain `Error` doesn't get mapped to the right HTTP status by tRPC and
  shows up to the client as `INTERNAL_SERVER_ERROR`.

### Tests
- `routers-clients-procedures` ‚Äî relaxed the `create` call assertion to
  match the new random `portalToken` (regex shape, not value).
- `routers-additional-coverage` (Notifications) ‚Äî removed five tests that
  asserted the now-deleted "throws NOT_FOUND when organization missing" and
  silent-error-swallowing behaviours. Replaced with tests that verify
  org/user scoping is enforced via the `updateMany` where clause and that
  DB errors propagate instead of being swallowed.

Net: **1389 tests pass** (one consolidated test removed), typecheck clean,
no regressions.

## Roadmap

### Security

| # | Severity | Area | Fix |
|---|----------|------|-----|
| S1 | High | `/api/v1/auth.ts` ‚Äî in-memory rate-limit map | Replace with the Upstash limiter already in `src/lib/rate-limiter.ts`; per-instance counters mean a multi-region deploy lets an attacker get N√ó the budget. |
| S2 | High | `/api/v1/*` uses Supabase user bearer tokens | Build a first-class API-key system (hashed at rest, scoped, rotatable). Keys are the right primitive for machine-to-machine; user JWTs expire and aren't safe to embed in client integrations. |
| S3 | Med | Stripe webhook dedup is per-process | Move idempotency to a `WebhookDelivery(provider, externalId, processedAt)` table so a multi-instance deploy can't double-process. |
| S4 | Med | Resend webhook has no dedup | Same table ‚Äî keyed on `email_id`. |
| S5 | Med | Inngest functions have no idempotency keys | Use `event.id` (or a content hash of the trigger row) as the idempotency key so a retried run can't double-create invoices. |
| S6 | Med | Portal dashboard session does not pin IP / UA | The session row stores both fields but never validates them. Pin (with UA fuzzy match) for sessions older than ~24 h, or shorten `SESSION_DURATION_MS` from 30 d to 24 h. |
| S7 | Med | `INNGEST_SIGNING_KEY` is `.optional()` | Required in production. The SDK silently accepts unsigned requests if missing. |
| S8 | Med | `GATEWAY_ENCRYPTION_KEY` rotation has no documented procedure | Document in `CONTRIBUTING.md`; consider a `gateway:rotate-encryption-key` script that re-encrypts every `GatewaySetting.configJson`. |
| S9 | Low | `bcryptjs` cost factor of 12 | Bump portal passphrases to 13 in `clients.create/update`. ~50% more CPU per auth, negligible against the 10-attempt limiter. |
| S10 | Low | Markdown preview uses a regex parser | Already pre-escapes input and the recent XSS fix held up ‚Äî but a real lib (e.g. `markdown-it` with `html: false`) removes a class of foot-guns. |

## How to extend this audit

The five parallel passes that produced this report were:
1. **Security** ‚Äî auth, IDOR, webhooks, secrets, file upload paths.
2. **Code quality / dead code** ‚Äî type safety, duplication, naming.
3. **Performance** ‚Äî N+1, missing indexes, bundle, cache opportunities.
4. **Architecture** ‚Äî router shape, error handling, REST consistency, service
   boundaries.
5. **Frontend / a11y / React** ‚Äî hydration, idiom drift, large components.

Each pass produced findings with file:line citations. The agents *do*
hallucinate occasionally ‚Äî verify every claim against the source before
acting. Notable misses to double-check next time: agent line numbers can be
off by a few, and agents may report behaviour that was already fixed in
recent commits as if it were still broken.
